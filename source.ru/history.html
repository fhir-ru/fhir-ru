История версий

Это разрабатываемая версия FHIR. Единственные изменения, которые отслеживаются здесь, это изменения после публикации DSTU. Изменения, сделанные до этого момента, можно посмотреть в DSTU #1 Version History. Обратите внимание, что полную историю всех изменений можно получить через архивы HL7 gForge SVN.

Каким образом работает версионирование FHIR

Данный раздел является черновиком

Политика версионирования FHIR основана на семантическом версионировании, однако имеются и некоторые различия в связи с тем, что FHIR - это спецификация, а не API программного обеспечения.

Существует одна разрабатываемая версия FHIR. Она проходит циклы развития, управляемые HL7. Каждый большой цикл развития завершается официальным голосованием, после которого публикуется новая спецификация. В терминах контроля версий, каждая опубликованная спецификация является веткой дерева разработки, которая может потом сама получить дальнейшие изменения от HL7 в рамках поддержки спецификации.

К спецификации могут быть применены следующие виды изменений:

  • Несущественные изменения не вызывают изменений в совместимых приложениях. Например изменение разделов нумерации, исправление неработающих ссылок, стилей, опечаток, и разъяснения, которые не меняют смысл. Некоторые исправления могут оцениваться таким образом, чтобы не создавать каких-либо ожиданий изменений в совместимых приложениях
  • Существенные изменения, которые не ломают совместимые приложения. Они вносят новую функциональность - изменения в спецификации, которые создают новые возможности, но не делают существующие приложения не-совместимыми, если они не были изменены
  • Критические изменения, делающие ранее совместимые приложения несовместимыми

Обратите внимание, что следующие изменения, по определению, считается некритическими , хотя некоторые реализации (в том числе эталонные) могут быть не в состоянии обрабатывать такие изменения без ошибок:

  • создание новых ресурсов
  • добавление новых элементов к существующим ресурсам
  • определение новых типов данных
  • добавление в API новых операций

Кроме того, изменения примеров не считаются существенными. Делается все возможное, чтобы гарантировать их правильность, однако изменения примеров в спецификации не считаются существенными изменениями.

Каждая версия FHIR представляет собой строку, состоящую из 4 частей: publication.major.minor.revision (издание.версия.субверсия.ревизия).

publication
  • Увеличивается, когда HL7 публикует обновленную спецификацию FHIR, например DSTU или нормативную версию FHIR
  • HL7 планирует делать это каждые 1-2 года
  • У первого DSTU была версия 0
major
  • Увеличивается каждый раз, когда вносится критическое изменение
  • Когда происходит новая публикация, это число сбрасывается в 0 для публикации и в 1 для ветки разработки
  • Так как HL7 не вносит критические изменения в виде технических поправок к опубликованной спецификации, у этих версий FHIR номер версии всегда будет X.0.n.r
  • Поскольку разрабатываемая версия является предметом постоянного анализа, обсуждения, голосования и неоднократных изменений, в ней следует ожидать критические изменения
minor
  • Увеличивается каждый раз, когда вносится существенное изменение
  • Сбрасывается в 0, когда изменяется основная версия (major)
revision
  • Увеличивается каждый раз, когда в спецификацию вносится изменение
  • На данный момент это номер версии SVN (это позволяет любому реконструировать источник, из которого была построена версия)

Дополнительные примечания:

  • Changes to a formally published specification (except for minor publishing corrections, such as correcting broken links) are only made via announced technical corrections
  • The reference implementations have 2 versions - the version of the specification that they implement, and their own version. Consult the reference implementation documentation for policy regarding this version number
  • The specification published by the continuous integration service (http://hl7-fhir.github.io/) build may not conform to this version policy, but the versions published on the HL7 web site will (see Directory of published versions)
  • Первый DSTU был опубликован до принятия этих правил с версией v0.80-2286. Она была обновлена технически исправлением до версии 0.0.81.2382, чтобы соответствовать политике версионирования, 9 мая 2014

0.5.0

DSTU Ballot, May 2015

This version had extensive change as a result of the January Draft ballot, ongoing testing, and the open change proposals (over 800 gForge tasks). List here is a summary of the major changes to resource content, but this is only a small amount of the overall changes.

Enumerations

  • All spaces removed
  • Extensive content changes not noted here

New Data Types

  • unsignedInt
  • positiveInt
  • Signature
  • Meta

Changed Data Types

  • Coding - remove valueSet
  • Attachment - add creation
  • Identifier - replace label with type
  • Timing - major rework of content
  • ElementDefinition - add label, code, rename 'formal' to definition, rename synonym to alias, add language to mapping, remove conformance and isExtensible and replace with strength

New Resources

  • BodySite
  • Claim
  • ProcessRequest
  • ProcessResponse

Removed Resources

  • CarePlan2 -> collapsed into CarePlan
  • FamilyHistory -> broken up into FamilyMemberHistory
  • InstitutionalClaim, OralHealthClaim, PharmacyClaim, ProfessionalClaim, VisionClaim -> collapsed into Claim
  • Other - use Basic instead
  • PendedRequest,Readjudicate, Reversal, StatusRequest, StatusResponse - use ProcessRequest/Response instead
  • SupportingDocumentation - use DocumentManifest instead

Renamed Resources

  • Alert -> Flag: 'alert' made people think it was an action like an alarm
  • SecurityEvent -> AuditEvent: it wasn't just for security purposes
  • ClinicalAssessment -> ClinicalImpression: people got confused with 'assessment' tools like APGAR score
  • Profile -> StructureDefinition: 'Profile' is the process, a package of statements
